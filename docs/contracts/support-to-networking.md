# Support to Networking Contract

This contract is the boundary between the Zenux support/helpdesk product and the
Zenux networking fabric.

Support owns the customer-facing ticket, consent, operator workflow, and
evidence bundle. Networking owns reachability, private access, provider adapter
execution, revocation, and stale-access checks.

## Request: `support.access.requested`

Support emits this request when a ticket needs private access or remote access
provider setup.

Required fields:

- `requestId`: stable idempotency key generated by support.
- `ticketId`: support ticket or case id.
- `customerId`: tenant/customer id.
- `authorizedContactId`: verified customer contact approving the session.
- `operatorId`: named engineer or automation actor.
- `deviceRef`: customer device reference, inventory id, or bootstrap code.
- `purpose`: `diagnostics`, `remediation`, `ssh-escalation`, `rdp`, `winrm`,
  `remote-desktop`, or `offboarding`.
- `requestedAccess`: list of required channels, ports, provider names, or
  capabilities.
- `expiresAt`: short-lived access expiry.
- `approvalRef`: Zenux approval id or policy decision id.
- `consentRef`: customer consent event id.

Optional fields:

- `preferredProvider`: `rustdesk`, `teamviewer`, `anydesk`, `beyondtrust`,
  `microsoft-remote-help`, `tailscale`, `cloudflare`, `forticlient`,
  `paloalto`, or `customer-owned`.
- `constraints`: customer policy constraints such as no unattended access, no
  file transfer, recording required, or customer engineer present.
- `networkHints`: known hostname, tailnet tag, tunnel name, subnet, or DNS zone.

Support must not pass secrets in this request. Secrets stay in a managed secret
store or the provider's secret store.

## Response: `networking.access.ready`

Networking emits this response when access is ready or when a provider-specific
launch path exists.

Required fields:

- `requestId`
- `accessId`: networking-owned access lifecycle id.
- `provider`: selected provider or transport.
- `sessionRef`: provider session id, tunnel id, route id, or machine id.
- `customerInstructionsRef`: support-renderable instruction template id.
- `operatorInstructionsRef`: operator launch or connection instruction id.
- `expiresAt`
- `revocationRequired`: always true for customer support access.

Optional fields:

- `connectionTarget`: hostname, route, private IP, or provider launch code.
- `auditRefs`: provider audit-log handles.
- `recordingRefs`: provider recording metadata handles, not raw recordings.

## Response: `networking.access.denied`

Networking emits this response when it cannot safely create access.

Required fields:

- `requestId`
- `reasonCode`: `missing-approval`, `unsupported-provider`,
  `device-unreachable`, `policy-denied`, `secret-missing`, or
  `provider-error`.
- `humanSummary`: short operator-facing explanation.
- `retryable`: boolean.

## Command And Channel Rules

- Shell access is escalation-only and must include `approvalRef`.
- Unattended access must be explicit in `constraints` and customer policy.
- Customer-to-customer access must remain impossible.
- Access expiry must be short-lived by default.
- Every access lifecycle event must be mirrored into Zenux security evidence.

## Offboarding Request: `support.access.revoke_requested`

Required fields:

- `requestId`
- `ticketId`
- `accessId`
- `operatorId`
- `reason`: `ticket-closed`, `customer-request`, `expired`, `incident`, or
  `test-cleanup`.

Networking must respond with `networking.access.revoked` or
`networking.access.revocation_failed`.
